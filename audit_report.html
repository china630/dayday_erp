DayDay ERP — отчёт по модулям

Сопоставление кода в apps/api, apps/web, packages/database с PRD.md (§4 модули 1–9, §7 платформа) и TZ.md (§2–§10). Ниже — отдельная оценка по каждому модулю (процент + зона + комментарий). Статический обзор; нагрузочные испытания не включались.

Зелёная — ≥ 75% (Ready) Жёлтая — 40–74% (Polish) Красная — < 40% (Missing)

Контекст документов

Нумерация модулей 1–9 в PRD §4 совпадает с TZ §2–§10. Платформенные релизы v5.6–v6.0 (изоляция тенантов, целостность, RBAC, инвентаризация, атомарность) задают сквозные правила для API и склада.

Недавние доработки (к этому аудиту): проведение InventoryAudit в одном prisma.$transaction с adjustStockInTransaction (PRD §7.10–7.11, TZ §10.1 v6.0); GET /inventory/audits и GET …/:id; тест inventory-audit.service.spec.ts. PRD §7.8 дополнен явным forbidNonWhitelisted.

Оценка по модулям (PRD §4 / TZ §2–§10)

Процент отражает покрытие требований модуля кодом, guards/политиками и наличием UI там, где модуль пользовательский.

Модуль 1 — IAM & Multi-tenancy

PRD §4.1 · TZ §2

93%

JWT + org в контексте, refresh cookie, M2M membership, switch org, VÖEN access request / invite, Prisma tenant extension + ALS, forbidNonWhitelisted, policy-модуль для финансовых ролей.

Модуль 2 — Ledger & Finance Core

PRD §4.2 + §4.11 (взаимозачёт) · TZ §3

92%

План счетов АР (seed), postTransaction, баланс Дт=Кт, закрытие периодов, быстрый расход с RolesGuard, NettingService (Дт 531 / Кт 211, InvoicePayment) с политикой роли.

Модуль 3 — CRM & Counterparties

PRD §4.3 · TZ §4

85%

Справочник контрагентов, VÖEN, роли клиент/поставщик; проводки 211/531 в связке с продажами и банком. Полная строгость «обязательных полей» и справочников банка — зона полировки.

Модуль 4 — Sales & Invoicing

PRD §4.4 · TZ §5

91%

Товары, инвойсы, НДС MVP, статусы и частичные оплаты, PDF, политика USER / PAID; цифровая подпись (задел v3). UX loading/toast не на каждой форме.

Модуль 5 — Cash Management

PRD §4.5 · TZ §6

83%

Мультивалютность, ЦБА, импорт CSV/Excel, матчинг, Direct Banking + модуль banking_pro; пилоты банков АР и «идеальный» клиент-банк — roadmap v3.

Модуль 6 — HR & Payroll (АР)

PRD §4.6 + §4.9 оргструктура · TZ §7

88%

Сотрудники, ГПХ, отпуска/больничные, ведомости, расчёт удержаний АР, проводки ЗП; департаменты и штат; BullMQ для крупных run. Полный кадровый контур уровня ЗУП — не цель MVP.

Модуль 7 — Reporting

PRD §4.7 · TZ §8

90%

ОСВ, P&L, баланс, cash flow; дебиторка, aging, сверка и UI взаимозачёта; tax-export; закрытие периода под Owner/Admin. Прямой обмен с e-taxes — v3.

Модуль 8 — Неизменяемый аудит

PRD §4.8 · TZ §9

87%

Перехватчик мутаций, old/new JSON, SHA-256, проверка целостности, BullMQ-архив; UI настроек аудита. «Только сторно» для проводок — позже по PRD.

Модуль 9 — Inventory (склад)

PRD §4.10 · TZ §10

91%

Склады, остатки, движения, закупка/трансфер, adjustStock с проводками; InventoryAudit DRAFT/APPROVED, атомарное проведение (§7.11), GET списка и карточки; UI /inventory/audit/new; unit-тест сервиса. Партионный FIFO как в v2 — roadmap.

Среднее по модулям 1–9 (арифметическое): ≈ 89%. Миграции БД: 26 файлов в packages/database/prisma/migrations. Автотесты API: 5 файлов (accounting, netting, tenant mergeWhere, stock, inventory-audit).

Вне нумерации 1–9 (дорожная карта)

Экосистема v3 (PRD §6 / TZ §13)

37%

Частично: VÖEN lookup, ЭЦП, заделы e-taxes и банка. Нет полного закрытия критериев приёмки v3.

Коммерция v4 + Super-Admin (PRD §7.1–7.6 · TZ §14–15)

84%

Подписка, квоты, READ_ONLY, gating модулей, биллинг, /super-admin. Полный биллинг-продакшн и мониторинг очередей — полировка.

Расширения v2: ОС, производство, Multi-GAAP (PRD §5 · TZ §12)

78%

Реализованы модули с gating; часть возможностей §5 (акты сверки PDF, FIFO) — не полностью.

Краткий Gap Analysis

  • Расширить автотесты по модулям 3–7 и e2e web.
  • Завершить интеграционный контур v3 (e-taxes, банки, ЭЦП в проде).
  • Единый UX (Empty State / disabled / toast) на всех формах модулей 4–5.
  • В §13 истории PRD при желании явно добавить строку v6.0 для симметрии с телом §7.11.

Вердикт: готовность к production

8.5 из 10

Модули 1–9 в сумме находятся в зелёной зоне; критичный пробел остаётся за v3. Атомарность инвентаризационной описи и расширение API списка/детали закрывают ключевые риски предыдущего аудита. Для оценки 9+ нужны стабильные внешние интеграции и более плотное тестовое покрытие по всем модулям.
