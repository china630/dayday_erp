## 1. Видение и ценностное предложение

### 1.1. Холдинг и консолидация валют отчётности

**Единая валюта отчётности:** при просмотре **сводных** показателей по **холдингу** (агрегация нескольких организаций — см. §4.12) денежные суммы приводятся к **базовой валюте холдинга** (по умолчанию **AZN**) по **официальному курсу ЦБА (ARMB)**.

**Стандарт точности (сводный P&L по холдингу): Monthly Slices (помесячная нарезка).** Период отчёта `[dateFrom, dateTo]` разбивается на **непересекающиеся календарные фрагменты внутри месяцев** (UTC). Для каждого фрагмента считается чистая прибыль организации за этот подпериод и переводится в валюту холдинга **`convert(amount, orgCurrency, holdingBase, fxAsOf)`**, где **`fxAsOf`** — **последний день фрагмента** (конец «куска» в пределах месяца). Суммы по фрагментам **складываются**. Отдельно: **дашборд cash/bank** по холдингу может использовать **одну дату `asOf`** для сравнения «на дату» — это не отменяет политику Monthly Slices для **сводного P&L за интервал**. Реализация: `HoldingsReportingService.consolidatedProfitAndLoss`, `CurrencyConverterService` — см. [TZ.md](./TZ.md) §1.1.

**Иерархия холдинга:** организации могут быть объединены в холдинг через `Organization.holdingId` (nullable). UI показывает структуру **Holding → дочерние компании** и отдельную секцию «свободных компаний» (не входящих в холдинг). Внутри продукта предусмотрен отдельный дашборд `/holding` с агрегированными показателями по холдингу (cash/bank в базовой валюте) и модальным созданием холдинга (кнопка «Новый холдинг»).

**Направление:** от надёжного ядра двойной записи и модулей учёта — к платформе, конкурентоспособной с классическими ERP за счёт скорости, соответствия локальным и при необходимости международным стандартам (МСФО), интеграций с госсервисами и банками Азербайджана.

| Персона | Ценность |
|---------|----------|
| **Бухгалтер** | «Закрытие месяца за 10 минут» (автосверки, амортизация, шаблоны для e-taxes) — целевое состояние дорожной карты |
| **Владелец** | Прозрачная дебиторка: кто, сколько и как долго должен |
| **Инвестор / управление** | Отчётность по мировым стандартам при параллельном учёте (Multi-GAAP) — как опция |
| **Администратор SaaS (v4)** | Предсказуемые тарифы, лимиты и оплата без «тихих» превышений квот |

**Core MVP** — модули **1–7** (PRD §4.1–4.7); **модуль 8** (аудит) — §4.8; **модуль 9** (склад/инвентаризация) — §4.10; дорожная карта и платформа — **§5–§7**. Полная детализация под все продукты 1С до старта кодирования не делается.

## 3. Архитектурные принципы

### 3.1. Изоляция данных (Multi-tenancy)

- Таблицы (кроме системных справочников) содержат `organizationId` (UUID).
- Запросы к API проходят через middleware / interceptor: из JWT извлекается `organizationId`, выборка ограничена текущей организацией.
- Глобальный фильтр Prisma: `findMany` / `findUnique` с `where: { organizationId }` (см. [TZ.md](./TZ.md) §2).

### 3.2. Модель доступа: «Пользователь — Организация» (Many-to-Many)

**Цель:** одно физическое лицо может управлять несколькими бизнес-единицами или работать в них под едиными учётными данными.

- **Изоляция identity:** профиль пользователя (email, пароль) отделён от данных организаций.
- **Владение (ownership):** при создании организации пользователь автоматически получает роль **OWNER**.
- **Присоединение к компании:** поиск существующей организации по **VÖEN**; если она уже зарегистрирована, отправляется **запрос на доступ (Access Request)** владельцу; владелец получает уведомление и может подтвердить или отклонить запрос, назначив роль.
- **Интерфейс переключения:** глобальный **селектор организаций (Switcher)** в UI, позволяющий менять контекст без повторного логина.

### 3.3. Финансовый движок (Double-Entry Bookkeeping)

**План счетов (Accounts)** — дерево; атрибуты: `code`, `name`, `type` (Asset, Liability, Equity, Revenue, Expense).

**Транзакция** — группирующая сущность: дата, описание, ссылка на документ-основание.

**Проводка (Journal Entry)** — `accountId`, `debit`, `credit`.

**Инвариант:** в одной транзакции Σ Debit = Σ Credit; финансовые операции в БД-транзакциях (ACID).

### 3.4. Интеграции (v3+, ориентир)

- Интеграции выносятся в адаптеры / отдельные границы; при падении внешнего API ядро остаётся доступным, интеграции показывают статус «недоступно».
- Тяжёлые задачи: **BullMQ + Redis** (ЗП, импорт, синхронизация банков, массовые операции).

## 4. Продуктовые модули (Core MVP и смежные возможности)

Нумерация модулей **1–9** совпадает с [TZ.md](./TZ.md) (§2–§10). Расширения дорожной карты — в §5–§7.

### 4.1. Модуль 1: IAM & Multi-tenancy

- Связь **пользователь — организация** в модели **many-to-many**; переключение активной организации без повторного входа (см. §3.2).
- Регистрация организации и администратора; при необходимости — запрос доступа к уже существующей организации по VÖEN.

### 4.2. Модуль 2: Ledger & Finance Core

- План счетов; при создании организации — **seeding** по единому плану АР (текущее состояние: JSON / скрипты в репозитории; целевое — копирование из **системных шаблонов** в БД, см. **§5.1**).
- Ручные операции (Journal Vouchers); автоматические — от других модулей.
- Валидация баланса, атомарность, закрытие периодов (детали — TZ).
- Главная книга; сводные отчёты — модуль 7.

### 4.3. Модуль 3: CRM & Counterparties

- Справочник: VÖEN, контакты; тип Клиент / Поставщик; физ / юр лицо; обязательные поля (TZ).
- **MDM (глобальный реестр):** единая запись на платформе по VÖEN (`GlobalCounterparty.taxId`, unique) для устранения дублей и консолидации данных. Локальные контрагенты организаций содержат ссылку `Counterparty.globalId` и сохраняют только организационно-специфичные поля (договоры/менеджер/заметки и т.п.).
- **Lookup при вводе VÖEN:** UI сначала делает поиск в глобальном реестре и автозаполняет наименование/адрес/НДС-статус; при отсутствии записи допускается внешняя проверка (e-taxes) с последующим созданием/обновлением записи в MDM.
- Взаиморасчёты: счета **211**, **531**; баланс контрагента при проводках.

| Аспект | Описание |

#### 4.5.2. Контроль учётных периодов и защита от «заднего числа»

**Финансовая дисциплина:** запрет на **проведение**, **редактирование** и **удаление** финансовых документов кассы (**MKO / MXO** и связанные проводки) в **закрытых** учётных периодах (согласовано с закрытием месяца в модуле отчётности и флагом блокировки проводок).

**Заднее число (backdating):** любая операция, которая приводит к **отрицательному остатку денежных средств** на счёте кассы (**101\***) на **любую** дату в интервале от даты документа до «сегодня», должна **блокироваться** с понятным сообщением пользователю («кассовый разрыв» на конкретную дату). Техническая реализация — валидация в `CashOrderService` при проведении расходного ордера (см. [TZ.md](./TZ.md) §6.0).

### 4.6. Модуль 6: HR & Payroll (локализация АР)

- Карточки сотрудников (ФИН, договор).
- Расчёт ЗП по правилам АР: подоходный, DSMF, ITS, безработица; ведомости; авто-проводки (TZ).

#### 4.6.1. Учёт отсутствий и справочник причин (ТК AР)

- **Справочник `AbsenceType` по организации:** статьи на азербайджанском (**Əmək məzuniyyəti** основной/дополнительный, **Sosial məzuniyyət**, **Ödənişsiz məzuniyyət**, **Təhsil və yaradıcılıq məzuniyyəti**, **Xəstəlik vərəqəsi**); у каждой строки — **`isPaid`**, поле **`formula`** (привязка к расчёту в Payroll/калькуляторах).
- **Запись отсутствия `Absence`:** период + ссылка на тип; при **ödənişsiz** — лимит календарных дней по справочнику (напр. 30); синхронизация с табелем без начислений для неоплачиваемых типов.
- **Формулы:** трудовой/социальный/учебный (оплачиваемый) — средняя за 12 месяцев **÷ 30,4 ×** календарные дни; больничный — первые **14** дней за счёт работодателя по **ставке от стажа** (остальное — фонд, вне суммы ERP); калькуляторы и API — **[TZ.md](./TZ.md) §7.0**.

**Tax calculator (Azerbaijan, Private Non‑Oil sector) — v16.1**

- **Input:** monthly **gross** salary (AZN).
- **Output:** per-employee breakdown: **income tax**, **DSMF (worker + employer)**, **unemployment (worker + employer)**, **medical insurance İTS (worker + employer)**, and resulting **net**.
- **Income tax:** 0% up to 8000 AZN, **14%** on the excess.
- **DSMF (social insurance):**
  - **Worker:** **10%** on up to 200 AZN + **10 AZN** on the excess (private non‑oil preferential rule).
  - **Employer:** configurable per organization (**22% standard** vs **15% preferential**) via `Organization.settings`.
- **Unemployment:** **0.5%** worker + **0.5%** employer.
- **Medical insurance (İTS):** progressive scale (see **TZ §7.0** for the current brackets used by the engine).
- **Rounding:** every tax line is rounded to **2 decimals** (ROUND_HALF_UP), net is calculated from the rounded tax lines.

### 4.9. Организационный менеджмент (HR & Org Structure) — целевое развитие

*Статус: не входит в Core MVP модуля 6 в текущем виде; фиксируется как продуктовое направление для аналитики и управления.*

**Цель:** построить иерархическую модель компании для аналитики расходов и управления подчинённостью (в дополнение к кадровым карточкам и расчёту ЗП).

**Организационная структура:** древовидная иерархия подразделений (**департаменты → отделы** и при необходимости более глубокие уровни). У каждого подразделения — **руководитель (Manager)**; связь сотрудников с узлами дерева.

**Штатное расписание (Staffing Table):** реестр должностей внутри подразделений с указанием **количества ставок (vacancies / headcount)** и **вилки окладов** (min–max или коридор по должности).

**Связь с финансами:** каждое подразделение выступает как **центр финансовой ответственности (ЦФО)**. Расходы на зарплату и налоги в отчётах **P&L** и **Cash Flow** могут быть **сгруппированы по департаментам** (и при необходимости детализированы по ЦФО).

**Контроль найма:** система **блокирует** создание (или активацию) карточки сотрудника на выбранной должности в подразделении, **если по соответствующей строке штатного расписания не осталось свободных ставок** (фактическое число занятых позиций ≥ утверждённому лимиту), либо выводит предупреждение — политика продукта (жёсткий стоп / мягкое согласование).

### 4.10. Модуль 9: Складская коррекция и инвентаризация (целевое развитие)

**Инвентаризация:** документ фиксации **фактических остатков** по номенклатуре (количество по складам/партиям — по политике учёта). **Сравнение** с **системными** остатками; вывод **разницы** (излишки / недостачи) для утверждения ответственным лицом.

**Оприходование / списание:** по результатам инвентаризации (или иной складской коррекции) — **автоматическое формирование финансовых проводок** с отнесением на счета **прочих доходов** и **прочих расходов** (и при необходимости на счета ТМЦ / себестоимости) в соответствии с утверждённой методикой отражения отклонений; суммы согласуются с оценкой товара (средняя себестоимость, партионный учёт — по настройкам модуля).

#### 4.10.1. Производство (Manufacturing): брак, отходы и побочный продукт

**Учёт потерь:** в спецификациях (**ProductRecipe** / строки рецепта) и в актах **выпуска готовой продукции** поддерживаются:

- **Коэффициент технологических потерь** (`wasteFactor` на строку компонента): увеличивает расход сырья относительно номинала рецепта (**фактический списание = норма × (1 + wasteFactor)**).
- **Побочный продукт** (целевое развитие): отдельная позиция номенклатуры / складской приход при частичном возврате материала в брак с отдельным учётом — по дорожной карте после MVP потерь.

Автоматическое **списание сырья** при выпуске ГП должно учитывать `wasteFactor`; расхождение «затраченное сырьё − выход ГП» отражается складскими движениями и при необходимости отдельным документом **списание отходов** (см. [TZ.md](./TZ.md) §10.2).

### 4.11. Взаимозачёт (схлопывание встречных долгов)

Один контрагент может одновременно фигурировать как **дебитор** и как **кредитор** организации:

| Счёт | Смысл |
|------|--------|
| **211** | Долг контрагента перед нами (дебиторка; непогашенная выручка по инвойсам). |
| **531** | Наш долг перед контрагентом (кредиторка поставщику; проводки с привязкой `counterpartyId` на счёте 531). |

**Алгоритм:** по выбранному `counterpartyId` система вычисляет остатки по обеим «сторонам»; **максимальная сумма зачёта** = **min(остаток 211, остаток 531)** в выбранной книге (NAS/IFRS для 531). При подтверждении создаётся одна **транзакция** с проводкой **Дт 531 — Кт 211** на сумму зачёта; сумма дополнительно **распределяется по неоплаченным инвойсам** (строки `InvoicePayment` на ту же транзакцию, без дублирования проводок), чтобы **дебиторка** и **старение** в отчётах совпадали с ГК.

**Интерфейс:** акт сверки (`/reporting/reconciliation`) — виджет «Встречные требования» и кнопка формирования взаимозачёта (активна только при **обоих** положительных остатках); карточка инвойса — предложение **«Оплатить зачётом»**, если по контрагенту возможен зачёт и по счёту есть непогашенный остаток.

**Налоговое соответствие (НК АР):** акт взаимозачёта **приравнивается к оплате** для целей учёта расчётов с контрагентом. Если обе стороны сделки — **плательщики НДС** (признак в карточке контрагента / справочнике), система **предлагает** оформить налоговую **e-qaimə** (электронная счёт-фактура) на сумму зачёта или отражает НДС проводками по счетам **241** / **541** в соответствии с методикой (детали — [TZ.md](./TZ.md) §10.3).

*Детали API (`FinanceService`, `NettingService`) и UX — [TZ.md](./TZ.md), §3 (дополнение к модулю 2), §10.3 и §8 (отчёты).*

### 4.13. Management Reports: Cash Flow (Direct method)

**Goal:** provide a management Cash Flow report for owners/accountants based on actual cash movements (direct method), with drill-down by cash flow items.

**Inputs (filters):**

- Period: `dateFrom..dateTo` (UTC dates, inclusive).
- Source filter: specific **cash desk** (cash orders) and/or specific **bank source** (bank statements).

**Data sources:**

- **Cash desks:** проведённые кассовые ордера (**KMO** / **KXO**) со обязательной статьёй ДДС (`cashFlowItemId`).
- **Bank:** строки `BankStatementLine` со статьёй ДДС и **`origin` ∈ {`FILE_IMPORT`, `MANUAL_BANK_ENTRY`}** (строки синхронизации вебхуком и т.п. в этот отчёт **не** включаются). Дата попадания в период: **`valueDate`**, если задана, иначе **дата выписки** (`BankStatement.date`).

**Реализация (v25.0):** API **`GET /api/reports/cash-flow`** (`CashFlowService.getDirectCashFlow`), UI **`/reports/cash-flow`**.

**Grouping and presentation:**

- Group by **CashFlowItem** (code + name).
- Split into **Operating / Investing / Financing** sections (category inferred by item code: `CF-INV*`, `CF-FIN*`, otherwise Operating).
- For every line show **Inflow**, **Outflow**, and **Net**; totals per section and grand total for the period.

**Performance and integrity:**

- All sums are calculated on the **API** using exact decimal arithmetic.
- Heavy periods may be cached in Redis for short TTL (keyed by organization + params).

### 5.1. Системные шаблоны плана счетов и онбординг организации (v1.1 / v2.0)

**Цель релиза:** вынести «эталонный» план счетов и сопутствующие платформенные справочники (в т.ч. маппинг для Multi-GAAP) из цепочки внешних скриптов приложения в **первоклассные сущности БД** без `organizationId`, с контролируемым жизненным циклом и UI для платформенного администратора.

#### 5.1.1. Проблема (as-is)

| Аспект | Риск / ограничение |
|--------|-------------------|
| **Источник правды** | План счетов для новых организаций задаётся кодом/файлами (`seed.ts`, JSON, одноразовые скрипты), а не версионируемой моделью в БД платформы. |
| **Операционка** | Изменение стартового набора счетов требует релиза или ручного вмешательства в среду; нет единого аудируемого контура для «что получит следующая регистрация». |
| **Согласованность** | Логика онбординга размазана между приложением и внешними шагами; сложнее гарантировать атомарность «организация создана ⇔ план счетов полностью развёрнут». |
| **Multi-GAAP** | Маппинг NAS → IFRS (и аналоги) должен жить рядом с шаблоном счетов как платформенные данные, а не дублироваться в скриптах. |

#### 5.1.2. Архитектурное решение: системные таблицы-шаблоны

- **План счетов NAS (шаблон для онбординга):** эталон хранится в глобальной таблице **`ChartOfAccountsEntry`** (каталог: `code`, `name`, `accountType`, опционально **`parentCode`** для иерархии, `sortOrder`, `isDeprecated`). Отдельная сущность с именем **`TemplateAccount` в схеме не используется** — термин «шаблон счёта» в продукте = строка **`ChartOfAccountsEntry`**. Копирование в тенант выполняется **`seedChartOfAccountsForOrganization`** / **`syncAzChartForOrganization`**: при **непустом** каталоге источником правды является **БД**, иначе однократная загрузка из JSON пакета `database` и запись в каталог.
- **Multi-GAAP:** платформенный маппинг NAS ↔ IFRS — таблица **`TemplateIFRSMapping`** (как и ранее); мутации шаблонов IFRS — по политике **`SUPER_ADMIN`** (см. скрипты/админ-контуры в ТЗ).
- Записи шаблонных справочников **без** `organizationId` описывают **продукт по умолчанию**, а не данные клиента.
- **RBAC:** создание/изменение строк **`ChartOfAccountsEntry`** через API **`/api/admin/chart-template`** — только **`SUPER_ADMIN`** (`/super-admin`). Обычные роли организации к каталогу не допускаются. Онбординг читает каталог серверной логикой в транзакции создания организации.
- **Инвариант:** каталог отделён от операционных `Account` (у счёта организации есть `organizationId`); смешение на уровне строк запрещено конвенцией.

#### 5.1.3. Изменение бизнес-логики: онбординг организации (Organization Onboarding)

- При **создании** сущности **`Organization`** (регистрация новой компании на платформе) система **не опирается** на внешние по отношению к транзакции БД скрипты и не выполняет отдельный «пост-хук» приложения как единственный источник плана счетов.
- В **одной атомарной транзакции** с фиксацией организации и членства владельца выполняется **детерминированное копирование** эталона в тенантное пространство на уровне **Prisma `$transaction` + SQL** (или эквивалент: пакетные `INSERT … SELECT` с подстановкой нового `organizationId`, сохранением иерархии `parentId` и ссылок для маппингов).
- **Реализация (v23.0):** копирование каталога NAS (`chart_of_accounts_entries` + JSON) в счета организации и bootstrap Multi-GAAP выполняется методом **`OrganizationsService.provisionChartOfAccountsFromTemplate(tx, organizationId)`** внутри **той же** `prisma.$transaction`, что и создание организации и подписки (регистрация / создание org для существующего пользователя).
- Поведение должно быть **идемпотентно-ориентированным** в смысле целостности: при откате транзакции организация не остаётся «без счетов»; при успехе набор счетов (и связанных сущностей по скоупу фичи) соответствует актуальному шаблону на момент регистрации.
- Детали соответствия полей (коды, типы, порядок сортировки, признаки активности) и политика **версионирования** шаблона (если вводится) фиксируются в [TZ.md](./TZ.md) при проектировании миграции.

#### 5.1.4. Миграция и обратная совместимость

| Этап | Действие |
|------|----------|
| **Первичное наполнение платформы** | Однократно (или при пустых шаблонных таблицах) данные эталона загружаются в `Template*` из канонического источника (например экспорт текущего JSON / миграция данных из существующего `seed`). |
| **`seed.ts` (локальная разработка)** | Сохраняется как **удобный механизм поднятия dev-среды**: минимально необходимые системные пользователи, роли, конфиг; **первичное заполнение** `Template*` при чистой БД; не является обязательным путём онбординга **production**-организаций после внедрения §5.1. |
| **Production** | Новые организации получают план счетов через транзакционное копирование из **`ChartOfAccountsEntry`** (при непустом каталоге) либо из JSON-bootstrap при пустом каталоге (§5.1.3). |
| **Уже существующие организации** | Обратная совместимость: существующие `Account` не перезаписываются автоматически массовой миграцией, если не объявлен отдельный проект синхронизации; изменения шаблона влияют на **новые** регистрации, если явно не задокументирована политика «догоняющих» миграций для старых тенантов. |

#### 5.1.5. Требования к UI: панель платформенного администратора

- В рамках защищённого контура **`/super-admin`** добавлен раздел управления **глобальным каталогом NAS** (**`ChartOfAccountsEntry`**, UI «План NAS (шаблон)») и (по мере необходимости) **`TemplateIFRSMapping`** — не виден обычному тенанту.
- Функции MVP раздела: просмотр дерева/списка шаблонных счетов, создание, редактирование, деактивация (по политике продукта — жёсткое удаление vs soft-delete), валидация уникальности кодов в рамках шаблона.
- Любое сохранение в этом разделе доступно **только** при **`SUPER_ADMIN`**; действия логируются в **глобальном аудите** платформы (см. §4.8 / §7.6).
- Изменения в шаблоне **применяются к будущим** регистрациям организаций с момента сохранения; явное предупреждение в UI, что уже созданные организации не меняются автоматически (если не реализован отдельный сценарий раскатки).

#### 5.1.6. Критерии приёмки (acceptance criteria)

1. **Модель данных:** нетенантный каталог **`ChartOfAccountsEntry`** (+ при необходимости **`TemplateIFRSMapping`**); операционные счета — `Account` с `organizationId`.
2. **Безопасность:** `GET/POST /api/admin/chart-template` и UI super-admin отклоняют запросы без **`SUPER_ADMIN`** (403).
3. **Онбординг:** создание организации в одной транзакции с **`provisionChartOfAccountsFromTemplate`** / **`syncAzChartForOrganization`**; при непустом `ChartOfAccountsEntry` источник — БД.
4. **Согласованность:** откат транзакции при ошибке копирования; нет «пустой» организации без стартового плана в рамках фичи.
5. **Миграция / dev:** первичное наполнение каталога из JSON при пустой БД; `seed.ts` для dev сохраняется.
6. **UI Super-Admin:** редактирование каталога NAS в `/super-admin`; новые регистрации получают актуальный каталог без релиза кода.

## 7. Дорожная карта: v4 — Commercial SaaS & Monetization

Завершение коммерческой модели поверх ядра и (при наличии) интеграций v3 (подразделы ниже).

### 7.1. Модель подписки: Конструктор тарифов (v8.1, расширение v8.8) и **Hybrid LEGO (v10.0)**

**Отказ от жёстких тиров как основной сюжет:** маркетинговые «три коробки» (Starter / Business / Enterprise) не задают продукт; источник правды по ценам — **конструктор** в БД (см. v8.8). Поле **tier** в `OrganizationSubscription` сохраняется для демо, миграций и аналитики; **ENTERPRISE** по-прежнему означает **полный доступ ко всем модулям** без перечисления slug.

**Сводка v10.0 (согласована с §7.12):** тарификация **по каждой организации (VÖEN)** — база + модули-надстройки + платные квоты; **единый месячный счёт платформы** на владельца по всем его организациям — см. §7.12.4. Имена и цены модулей ниже — **ориентир для каталога `PricingModules`**, не константы в коде.

| Слой | Описание |
|------|----------|
| **Foundation (база платформы)** | Ежемесячная **базовая цена за активную организацию** (ориентир **29 AZN/мес.**) — доступ к **базовому учёту** (Ledger, CRM, дашборд) и **1 пользователь** (сид) в рамках этой организации; задаётся в Super-Admin (`SystemConfig`, ключ `billing.foundation_monthly_azn`). Дополнительные пользователи — через **квоты** (§7.12.3). |
| **Module Marketplace (LEGO)** | Модули включаются **по организации** независимо; цена в месяц — в **`PricingModules`**. Ориентир цен (AZN/мес., v10.0): **Kassa Pro 15**, **Banking Pro 19**, **Warehouse 25**, **Manufacturing 39**, **HR 19**, **IFRS 29** (ключи slug — как в каталоге, например `kassa_pro`, `banking_pro`, …). |
| **Quota Management** | Расширение лимитов (сотрудники, объём документов/инвойсов) — **цена за единицу надбавки**: например стоимость **за блок из 10 дополнительных сотрудников** и за **пакет документов**; значения в `SystemConfig` (`billing.quota_unit_pricing_v1`) и в подписке клиента (`customConfig.quotas`). |
| **Пакеты (bundles)** | Именованные наборы модулей с **скидкой пакета** (%) — таблица **`PricingBundles`**, настройка в Super-Admin (**Paket yaradıcısı**); для клиента показывается **предпросмотр** месячной и годовой суммы. |
| **Период оплаты** | При оплате **за год** — **автоматическая скидка** (по умолчанию **20%**) к итогу; процент хранится в `SystemConfig` (`billing.yearly_discount_percent`). |

**Формула итога (биллинг):** `Total = BasePrice + Σ(выбранные модули) + (доп. квоты × цена единицы)`; затем применяется **скидка пакета** (если выбран bundle), затем при годовом периоде — **годовая скидка**. Детали схемы — [TZ.md](./TZ.md) §14.

**Ориентир устаревших пакетов (маркетинг / миграция):** Starter / Business / Enterprise остаются **legacy-пресетами** в `SystemConfig` (`billing.price.*`) для совместимости; новые продажи ориентируются на конструктор и каталог `PricingModules`.

### 7.2. Продуктовые механизмы

- **Feature gating:** блокировка UI и API при неоплаченном модуле; 403 + код причины на API.
- **Subscription management:** текущий план, дата окончания, модули и квоты по организации — только для роли **Owner** (§7.12.1); привязка карты и просмотр истории платформенных платежей — также только **Owner**.
- **Quota enforcement:** блокировка создания записей при исчерпании лимита (политика: жёсткий стоп vs предупреждение).
- **Billing MVP:** платёжный шлюз (ориентиры: Pasha Bank, E-Pul); **единый месячный счёт** на владельца по портфелю org; идемпотентность webhooks; аудит смены плана.

| Принцип | Описание |
|--------|----------|
| **ORM-уровень** | Использование механизмов Prisma (`$extends` / Client Extension) для принудительной фильтрации по `organizationId` на операциях чтения/изменения/удаления по моделям с мультитенантностью. |

### 7.8. Платформа (v5.7): консистентность данных и UX (Data Integrity)

**Цель:** исключить частично применённые бизнес-операции, затрагивающие учёт и склад.

| Принцип | Описание |
|--------|----------|
| **«Всё или ничё»** | Любая операция, порождающая финансовый след (проводки NAS/IFRS, движения ТМЦ, статусы документов с проводками), выполняется в **одной** атомарной транзакции БД (`prisma.$transaction`), включая связанные обновления сущностей. |
| **Согласованность** | Если проводка или движение не созданы — изменения доменных данных в рамках этой операции не фиксируются (откат транзакции). |
| **API** | Глобальный `ValidationPipe` с **whitelist** и **`forbidNonWhitelisted: true`** снимает неизвестные поля DTO; входные DTO типизированы декораторами `class-validator` (согласовано с §7.9 и [TZ.md](./TZ.md) §2 / §17). |
| **UX** | Пустые таблицы — единый **Empty State**; ошибки мутаций — toast; кнопки «Сохранить» / «Провести» — disabled + индикатор загрузки. |

### 7.9. Платформа (v5.8): матрица полномочий (Advanced RBAC)

**Цель:** каждая бизнес-мутация (создание, изменение, удаление) проходит через фильтр политик доступа, разграничивающий права **бухгалтера**, **менеджера (роль User)** и **владельца/администратора**.

| Принцип | Описание |
|--------|----------|
| **Политики** | Для финансовых документов действуют явные правила: например, роль **User** не изменяет инвойсы в статусе **PAID** и не создаёт ручные проводки в журнале, доступные бухгалтеру и выше. |
| **Слой API** | Проверки ролей в сервисах и/или guards на контроллерах; критичные маршруты — только **OWNER**, **ADMIN**, **ACCOUNTANT** где требуется. |
| **Согласованность с ТЗ** | Детали — **Strict Validation** (`forbidNonWhitelisted`) и **Policy Guard** в [TZ.md](./TZ.md) (§2 и §17). |

### 7.10. Процессная инвентаризация (v5.9 — Final Consistency / RC1)

**Цель:** перейти от разовых ручных корректировок к документу **«Инвентаризационная опись»** с фиксацией излишков и недостач и проведением в один шаг.

| Принцип | Описание |
|--------|----------|
| **Сущность** | **InventoryAudit (1 склад)** + строки **InventoryAuditLine (N)**: `systemQty`, `factQty`, `costPrice`. Черновик формирует **снимок системных остатков** по выбранному складу; далее фиксируется факт и оценка расхождений. |
| **Проведение** | При **APPROVED** выполняется атомарное проведение в **одной** `prisma.$transaction`: создаются **проводки** по отклонениям (излишек: **Дт 201/204 — Кт 611**, недостача: **Дт 731 — Кт 201/204**) и **движения склада** (`StockMovement`, reason=ADJUSTMENT), после чего статус описи меняется на APPROVED (см. также TZ §10.1). |
| **Доступ** | Ручные проводки и проведение описи — по политикам RBAC (не ниже бухгалтера там, где задано в API). |

### 7.11. Платформа (v6.0): усиление атомарности (неделимый блок БД)

**Цель:** любой документ, порождающий связанные финансовые и складские эффекты (в частности **Инвойс**, **Инвентаризационная опись**, **Зарплата**), обязан завершать **все** связанные операции (проводки, движения ТМЦ, смена статусов с проводками) внутри **одного** неделимого блока транзакции БД (`prisma.$transaction` или эквивалент), без частичной фиксации при ошибке на любом шаге.

### 7.12. Billing & Subscription (v10.0) — владение, Hybrid LEGO, квоты, счета платформы

Дополнение к §7.1–7.2: **коммерческий контур** привязывается к **владельцу организации** и к **каждой организации (VÖEN)** как отдельному юниту оплаты. Детали схемы БД и API — [TZ.md](./TZ.md) §14.8.

#### 7.12.1. Иерархия владения (Ownership & Access)

| Понятие | Описание |
|--------|----------|
| **Account (User)** | Единая точка входа: один пользователь может состоять в **нескольких** организациях и **владеть** несколькими (как `ownerId` / роль **OWNER**). |
| **Owner** | Эксклюзивный статус в рамках организации: **только** пользователь с ролью **OWNER** управляет подпиской и коммерческими настройками по этой организации, видит **историю платежей** платформы по своему портфелю, может **привязать платёжное средство (карту)**. Роли **Admin**, **Accountant**, **User** к разделу **Billing** не допускаются (ни просмотр, ни мутации). |
| **Organization.ownerId** | FK на пользователя — владелец для **выставления счетов платформы** и **Transfer**; при создании организации `ownerId` = создатель; смена — только **Change Owner**. |
| **Transfer of Ownership** | Передача прав другому пользователю **внутри системы**: текущий и новый владелец подтверждают (**double opt-in**). В одной БД-транзакции обновляется `organizations.ownerId` и роли membership (например, прежний владелец → `ADMIN`, новый → `OWNER`). |

**Сводный биллинг:** пользователь, который является **OWNER** (и обычно `ownerId`) по одной или нескольким организациям, получает **один** агрегированный счёт платформы в месяц (§7.12.4).

#### 7.12.2. Юнит тарификации — Hybrid LEGO (по организации)

- **1 биллинг-юнит = 1 организация (1 VÖEN).** Фиксированных «коробочных» планов нет: набор модулей и квот задаётся **независимо для каждой** организации.
- **Формула месячной оценки по организации (вход в `EstimatedNextPayment` и строки счёта):**

\[
\text{Monthly}_\text{org} = \text{BasePrice} + \sum(\text{ActiveModule}_i) + \sum(\text{OverlimitQuota}_k)
\]

  - **BasePrice** — за организацию (ориентир **29 AZN/мес.**; факт — `SystemConfig` / `basePriceSnapshot`); включает **базовый учёт** и **1 пользователя** в рамках этой организации (§7.1).
  - **Active modules** — сумма цен включённых модулей из каталога; ориентиры цен — §7.1 (таблица Module Marketplace).
  - **Overlimit quotas** — доплата за превышение включённых лимитов (§7.12.3).

Каталог **`PricingModules`** — источник правды по ценам модулей; **`basePriceSnapshot`** на организации фиксирует базу на дату.

#### 7.12.3. Квоты (Scaling) — платные пакеты расширения

База Foundation даёт **одного пользователя** на организацию; остальное — **масштабирование** через платные пакеты (лимиты и цены — в `SystemConfig`, ключ `billing.quota_unit_pricing_v1`, и в `customConfig.quotas`).

| Ось | Описание |
|-----|----------|
| **Пользователи / сотрудники** | Дополнительные места (например блоки по **10** человек за **15 AZN** — ориентир) сверх включённого в базу **1** сида. |
| **Диск** | Ориентир: **1 GB** в базе; сверх — пакеты (например **5 AZN / 5 GB**). |
| **Инвойсы продаж (`maxInvoicesPerMonth`)** | Ориентир базы **100/мес.**; сверх — пакеты (например **500** шт., цена из конфига). |

Превышение не должно приводить к «тихому» списанию: блок / upsell / явное согласие на докупку (§7.2).

#### 7.12.3.1. Gating UX при достижении квот

**Поведение интерфейса:** при достижении лимита (**пользователи / сотрудники**, **объём склада**, **инвойсы за месяц** и др. квоты из §7.12.3) приложение **не должно «падать»** с необработанной ошибкой.

| Правило | Описание |
|--------|-----------|
| **Создание** | Кнопки и формы **создания** новых записей, которые увеличивают потребление квоты, **блокируются** или показывают **модальное окно** с предложением **апгрейда тарифа** / докупки пакета. |
| **Чтение** | Просмотр (**Read-only**) данных, отчёты и навигация по уже созданным объектам **сохраняются** (кроме режима READ_ONLY по подписке — §7.3). |
| **API** | Мутации, нарушающие квоту, возвращают **HTTP 402 Payment Required** с машиночитаемым кодом (см. [TZ.md](./TZ.md) §14.8.7); фронтенд перехватывает ответ и показывает **UpgradePlanModal** (или эквивалент). |

#### 7.12.4. Платежи и счета платформы (не путать с Sales Invoice)

**Счёт платформы** — документ к **владельцу** за подписку DayDay ERP. В коде и БД — отдельные сущности (**`billing_invoices` / `BillingInvoice`**), не таблица **`invoices`** (продажи).

| Правило | Описание |
|--------|----------|
| **Единый месячный инвойс** | **Owner** получает **один** счёт платформы за **месячный** период, **суммирующий** все начисления по **всем** организациям, где он владелец (`ownerUserId` = этот пользователь). Внутри счёта — строки по каждой организации (VÖEN) и типу начисления (база, модуль, квота). |
| **Billing History** | Раздел UI: **детализированная таблица** транзакций/счетов платформы (дата, период, сумма, статус); для каждого счёта — **скачивание PDF** (`pdfLink`). |

Обязательные атрибуты счёта:

| Атрибут | Смысл |
|--------|--------|
| Привязка к владельцу | `ownerUserId` — плательщик |
| Период | `periodStart`, `periodEnd` (месячный цикл) |
| Статус | черновик / выставлен / оплачен / просрочен |
| PDF | `pdfLink` на сгенерированный счёт |
| Строки | разбивка по организациям: `organizationId`, описание, `amount` |

Детализация полей и API — **TZ §14.8**.

#### 7.12.5. Платёжный цикл и pro-rata

- **Списание / выставление счёта:** раз в месяц (**Monthly Billing Cycle**), календарь и таймзона — зафиксировать (UTC или Asia/Baku).
- **Pro-rata:** при **включении модуля** в середине расчётного периода — доля к оплате пропорционально **оставшимся дням** периода (округление — правило бизнеса: до копеек AZN).
- При **отключении** модуля: либо до конца оплаченного периода доступ сохраняется, либо pro-rata-кредит — политика продукта (зафиксировать в ТЗ при реализации).

## 10. Нефункциональные требования

### 10.0. Журнал безопасности (роль OWNER)

| Требование | Описание |
|------------|----------|
| **Доступ** | Просмотр полного журнала аудита организации и детальных снимков изменений — **только роль OWNER** (отдельно от журнала «Аудит» для Admin/Owner в настройках, если политика продукта различает уровни). |
| **Данные** | Записи из `AuditLog`: кто (`userId` + ФИО), тип объекта, идентификатор, действие (CREATE/UPDATE/DELETE по маппингу HTTP-метода), время, при необходимости сравнение **oldValues / newValues**. |
| **Фильтры** | По пользователю, типу сущности, диапазону дат; постраничная навигация. |
| **Прозрачность** | На карточках ключевых объектов (например счёт-фактура, сотрудник) — вкладка **«История изменений»** с логами **только по данному `entityId`**. |
| **Производительность** | Запись аудита не должна блокировать ответ API дольше основной бизнес-операции: допускается асинхронная запись после успешной мутации (см. TZ §9). |

| Атрибут | Требование |
|---------|------------|
| **Надёжность** | ACID для финансовых операций |
| **Аудит** | `AuditLog` с `oldValues`/`newValues`, хеш SHA-256, IP/UA; архив >1 года; «только сторно» для проводок — позже |
| **Производительность** | Пагинация, индексы по `organizationId` + `createdAt` |
| **API** | OpenAPI (Swagger) |
| **Контейнеризация** | Docker Compose (dev/prod MVP) |
| **Очереди** | BullMQ для фоновых задач |
| **Файлы** | Абстракция `STORAGE_SERVICE`; **production:** S3-compatible по умолчанию (`STORAGE_DRIVER=s3`); локально допускается `local` (см. §11.0) |
| **Почта** | Nodemailer SMTP — инвойсы, восстановление пароля |
| **Интеграции (v3+)** | TLS 1.2+ к госорганам; секреты не в коде; логи подписей |
| **Биллинг (v4)** | Секреты шлюза в vault/env; идемпотентность webhooks |

### 10.1. UX / UI Standards (v8.9)

| Правило | Требование |
|--------|------------|
| **Контраст** | Все интерактивные элементы (кнопки, ссылки) должны иметь **чёткий контраст** с фоном **#EBEDF0**. Запрещено использовать бледные серые для **основных** действий; первичные кнопки — **#2980B9**, заголовки страниц — **#34495E** (см. [DESIGN.md](./DESIGN.md)). |
| **Иконография** | Каждому пункту бокового меню **строго назначена иконка** (Lucide); отсутствие иконки — нарушение стандарта. |
| **Границы** | Карточки и панели — обводка **#D5DADF** (`border-muted`), не «растворяющиеся» светло-серые рамки без контраста. |
| **Empty State** | Пустые списки — единый компонент с **иконкой**, заголовком и пояснением по центру. |

## 11. Критерии приёмки Core MVP

### 11.0. Профиль организации, объектное хранилище и метод оценки запасов (v19.0 / v25.0)

| Тема | Требование |
|------|------------|
| **Объектное хранилище (обязательно для приёмки прод-сборки)** | Логотипы организаций, PDF счетов и прочие вложения сохраняются через **`STORAGE_SERVICE`** с **`STORAGE_DRIVER=s3`** и настроенными переменными **`S3_*`** (bucket, region, credentials, при необходимости `S3_PUBLIC_BASE_URL`). Локальная разработка может использовать **`STORAGE_DRIVER=local`** и `STORAGE_LOCAL_ROOT` без облака. |
| **Профиль** | В карточке организации доступны: **юридический адрес**, **телефон**, **ФИО/наименование руководителя** (текст для печати), **логотип** (файл → URL в объектном хранилище), **VÖEN** и наименование (как сейчас). |
| **Банковские реквизиты** | Несколько счетов организации: **банк**, **номер счёта**, **валюта** (AZN / USD / EUR), **IBAN**, **SWIFT** — для документов и интеграций. |
| **Оценка запасов** | Параметр организации **`valuationMethod`**: **AVCO** (средневзвешенная по `stock_items.average_cost`, текущее поведение) или **FIFO** (себестоимость списания по «очереди» приходов из `stock_movements` типа IN). Выбор влияет на COGS при отгрузке (инвойс), списаниях/перемещениях и выпуске ГП (компоненты). |
| **Глобальный справочник** | По VÖEN ведётся строка в **`global_company_directory`**; при сохранении организации или контрагента данные **асинхронно** обновляются для Smart-fill. |

1. Пользователь может создать организацию.
2. Пользователь может создать два счёта (например, «Касса» и «Доход»).
3. При создании оплаченного инвойса создаётся транзакция: **Дебет (Касса) = Кредит (Доход)**.
4. Отчёт Trial Balance показывает корректные остатки по счетам.

## 12. Зафиксированные решения (скоуп, тенанты, валюты, периоды, локаль)

| Тема | Решение |
|------|---------|
| **Тенант** | 1 tenant = 1 юрлицо (VÖEN). Несколько компаний у пользователя = несколько организаций. |
| **Курсы валют** | Источник: XML/API ЦБА (cbar.az); обновление ежедневно в 00:01; переоценка валютных счетов — регламент в конце месяца. |
| **Закрытие периода** | Только **Owner** или **Admin**. |
| **Архивация (Soft Delete)** | **Полное удаление (Hard Delete)** организаций, контрагентов и **транзакций/проводок**, по которым есть связанные движения (склад, оплаты, аудит), **запрещено**. Вместо этого — **архивация**: флаги `isDeleted` / `deletedAt` и скрытие из списков по умолчанию (см. [TZ.md](./TZ.md) §16). |
| **Интеграции MVP** | Импорт CSV/Excel; API банков — после sandbox/доступов. |
| **UI и i18n** | Язык по умолчанию **RU**; архитектура **i18next**, готовность к **AZ**. |
| **Время и форматы** | В БД — **UTC**; отображение — локаль Азербайджана, суммы в **AZN**. |
| **Локальная инфра (Windows)** | Данные контейнеров, файлы, npm-кэш, temp — **D:** (`D:\DockerData\dayday_erp`); образы Docker — хранилище на D в Docker Desktop. |
